<p class="section-eyebrow">§ 1 · Premise</p>
## Transformer-XL's limitation

Transformer-XL (Dai et al. 2019) gave decoder models a sliding short-term memory: each chunk
could attend to the previous chunk's activations as additional context, recurrently. This
extended effective context but capped at one chunk's worth of past — older activations were
discarded entirely.

The Compressive Transformer's question: rather than discarding evicted activations, can we
*compress* them into a smaller representation that still carries usable signal?

<p class="section-eyebrow">§ 2 · Two memory tiers</p>
## Sliding cache + compressed long-term memory

Each transformer layer maintains two memories:

- **Short-term memory** $M$: a FIFO of the last $N_s$ token activations, exactly as in
  Transformer-XL.
- **Long-term compressed memory** $\tilde{M}$: when $N_s$ activations are evicted from $M$,
  they are passed through a compression function $f_c$ that reduces them to $N_s / c$
  activations, where $c$ is the compression ratio (typically 3 or 4). The output is prepended
  to $\tilde{M}$.

Attention at layer $\ell$ then runs over the current chunk's queries, attending to keys/values
from the current chunk, $M$, and $\tilde{M}$ — three memory regions of decreasing temporal
resolution.

$$\mathrm{Attn}_t = \mathrm{softmax}\!\left(\frac{\mathbf{q}_t \cdot [K_{\text{current}}; K_M; K_{\tilde{M}}]^\top}{\sqrt{d_h}}\right)[V_{\text{current}}; V_M; V_{\tilde{M}}].$$

```python
def compressive_layer(x, attn, short_mem, long_mem, f_compress, c=4):
    # When short_mem overflows, compress the oldest c tokens into one
    # summary vector and prepend to long_mem.
    if len(short_mem) > N_s:
        evicted = short_mem[:c]                  # oldest tokens leave first
        short_mem = short_mem[c:]
        compressed = f_compress(evicted)         # [c, d] -> [1, d]
        long_mem = concat([long_mem, compressed])
    keys = concat([x, short_mem, long_mem])
    return attn(x, keys, keys)
```

<p class="section-eyebrow">§ 3 · The compression function</p>
## What works as f_c

Rae et al. evaluate four compression functions (Table 3):

1. **Mean pooling**: collapse $c$ activations by averaging.
2. **Max pooling**: collapse by per-dimension max.
3. **1D convolution**: a learned stride-$c$ conv.
4. **Attention compression**: a small attention block that summarizes the $c$ activations.

Empirically: attention compression wins on language tasks (lowest perplexity), 1D conv is a
close second, mean/max pooling lose by a noticeable margin. The intuition is that the
information density of language activations is uneven across the $c$ tokens — pooling treats
all positions equally, while attention can pick out the informative ones.

<p class="section-eyebrow">§ 4 · Results</p>
## Empirical evidence

On PG-19 (a long-document language benchmark released alongside the paper), Compressive
Transformer with attention compression achieves 33.6 perplexity at sequence length 4K plus 1K
short-term plus 1.5K compressed memory — vs 36.3 for Transformer-XL with equivalent effective
context (Rae et al. Table 5). The improvement compounds with longer documents.

The technique didn't see direct production adoption as 2020–22 long-context efforts pivoted
toward other approaches (ALiBi, RoPE rescaling, sliding window). But its core idea —
*compression rather than eviction* — is what Activation Beacon and similar modern recipes
revisit at training-time scale. See the [Activation Beacon entry](/llm-arch-kb/long-context/activation-beacon/)
for the descendant that won.

<Summary>
**Compressive Transformer in one breath.** Two-tier memory: short-term sliding cache plus a

that compression is a viable alternative to eviction; Activation Beacon and similar 2024
techniques are its modern descendants.
