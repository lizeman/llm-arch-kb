<p class="section-eyebrow">§ 1 · Premise</p>
## RoPE assumes a per-head K vector

Standard RoPE rotates each head's $\mathbf{q}$ and $\mathbf{k}$ vector by an angle that depends
on token position. The rotation is applied **after** the per-head linear projections and
**before** the dot product:

$$
\text{attn}(t, s) = \bigl(R_t \mathbf{q}_t\bigr)^{\!\top}\!\bigl(R_s \mathbf{k}_s\bigr) =
\mathbf{q}_t^{\!\top}\, R_{s-t}\, \mathbf{k}_s.
$$

The cached object is the rotated $\mathbf{k}_s$ — one full per-head K vector per token per
layer. That cache is what makes long-context inference expensive, and what
[MLA](/llm-arch-kb/attention/mla/) tries to compress.

MLA caches a small **position-free** latent $\mathbf{c}^{KV}_s \in \mathbb{R}^{d_c}$ instead of
per-head K. To reconstruct K at attention time, the implementation absorbs the up-projection
$W_{UK}$ into $W_Q$ — a clean trick that removes the up-projection from the inner loop. RoPE
breaks that trick.

<p class="section-eyebrow">§ 2 · The two non-options</p>
## Why you can't just bolt RoPE onto MLA

**Apply RoPE after the up-projection.** Keep the latent position-free; rotate per-head K each
time you read it. The rotation matrix $R_s$ depends on the absolute key position, so it can't
be folded into $W_Q$ ahead of time. You pay the up-projection at every attention step — losing
the main inference advantage of compression.

**Apply RoPE before the down-projection.** Rotate $\mathbf{h}_s$ first, then compress to
$\mathbf{c}^{KV}_s$. Now the latent is position-aware — but the rotation is entangled with
$W_{DKV}$ in a way that can't be inverted at attention time. The query side would need to undo
the rotation through a non-orthogonal projection, which is a different (and broken) operation
from the standard RoPE inner-product identity.

Either way, you lose either compression or compatibility.

<p class="section-eyebrow">§ 3 · The decoupled head</p>
## Carry position in a separate small slot

Decoupled RoPE splits the K projection into two parts:

- A **position-free body** of dimension $d_h - d_R$ per head, reconstructed from the latent.
- A **position-aware head** of dimension $d_R$ (a single shared head, not per-head),
  projected directly from the unrotated input and rotated by RoPE.

$$
\mathbf{k}_s = \bigl[\, \underbrace{W_{UK}\,\mathbf{c}^{KV}_s}_{\text{from latent}} \,;\,
\underbrace{R_s\bigl(W_{KR}\,\mathbf{h}_s\bigr)}_{\text{decoupled RoPE head}} \,\bigr]
$$

The query is split symmetrically:

$$\mathbf{q}_t = \bigl[\, W_{Q,\text{body}}\,\mathbf{h}_t \,;\, R_t\bigl(W_{Q,\text{rope}}\,\mathbf{h}_t\bigr) \,\bigr]$$

so the dot product splits cleanly into a position-free part (which uses the absorbed
$W_Q W_{UK}$) and a position-aware part (which uses the standard RoPE identity on the small
$d_R$-wide slot).

```python
# Per-token, per-layer
c_kv   = h @ W_DKV                        # [B, T, d_c]   compressed latent
k_body = c_kv @ W_UK                      # [B, T, H, d_h - d_R]  reconstructed at attention
k_rope = rope(h @ W_KR, pos)              # [B, T, d_R]           shared, rotated, cached

q_body = h @ W_Q_body                     # [B, T, H, d_h - d_R]
q_rope = rope(h @ W_Q_rope, pos)          # [B, T, H, d_R]

logits = (q_body * k_body).sum(-1) + (q_rope * k_rope).sum(-1)
```

DeepSeek-V2 sets $d_R = 64$ — small relative to a typical $d_h = 128$, large enough to carry
useful position information. The $k_{\text{rope}}$ slot is **shared across all heads at that
layer**, so it costs $d_R$ floats per token per layer in cache — additive on top of the latent
$d_c$ floats.

<p class="section-eyebrow">§ 4 · Cache accounting</p>
## What you actually store

For a layer with $H$ heads, head dim $d_h$, latent dim $d_c$, and decoupled RoPE dim $d_R$:

| Scheme | Cache per token per layer |
|---|---|
| MHA | $2 \cdot H \cdot d_h$ |
| GQA ($G$ groups) | $2 \cdot G \cdot d_h$ |
| MQA | $2 \cdot d_h$ |
| MLA + decoupled RoPE | $d_c + d_R$ |

DeepSeek-V2 uses $H = 128$, $d_h = 128$, $d_c = 512$, $d_R = 64$ — so the cache per layer is
$512 + 64 = 576$ floats, vs $2 \cdot 128 \cdot 128 = 32{,}768$ for MHA. That's the source of
the ~5–7× end-to-end cache reduction in V2's Table 9 (which also accounts for the up-projection
absorption).

<p class="section-eyebrow">§ 5 · Generalization</p>
## A pattern, not just an MLA detail

Decoupled RoPE is the general recipe for combining RoPE with **any** compressed-cache scheme.
The same idea — pull out a small position-aware K slot, leave the rest of the cache
position-free — applies to:

- Other low-rank K, V compressions explored in 2024–2025 follow-ups.
- Cache-mixing schemes that share K across layers but want per-token position information.
- Hybrid attention layers that combine a compressed long-range view with a small position-aware
  short-range view.

The cost is a constant additive $d_R$ floats per token per layer. The benefit is that the
compressed body retains the inference-time absorb-into-$W_Q$ trick.

<Summary>
**Decoupled RoPE in one breath.** Pull a small $d_R$-wide K head out of the position-free
latent, rotate that head with RoPE, leave the rest of the cache RoPE-free. Lets MLA-style
compression keep the absorb-$W_{UK}$-into-$W_Q$ inference trick. DeepSeek V2 / V3 use
$d_R = 64$, shared across heads.
</Summary>
